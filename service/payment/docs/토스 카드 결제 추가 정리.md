# toss 카드 결제 브런치 정리

## 왜 이 변경이 필요했는가

기존 payment 모듈에는 Toss 연동이 이미 있었지만, 그 흐름은 `charge` 기반의 예치금 충전용이었다.

- 기존 구조
  - `POST /api/payments/confirm`
  - charge 승인 성공 시 wallet 증가
  - wallet_transaction 기록
- 이번 요구사항
  - 주문 카드 결제
  - wallet_transaction 미사용
  - order 생성 이후 프론트가 Toss 위젯을 호출하고, success 결과를 payment가 최종 승인 처리
  - 부분 취소/환불을 고려한 append-only 거래 이력 필요

그래서 기존 `charge`를 재활용하지 않고, 주문 카드 결제 전용 거래 이력 모델과 confirm API를 별도로 추가했다.

설계 포인트:

- 승인 시 `orderItem` 단위 row를 여러 건 저장한다.
- 나중에 취소/환불도 기존 row 수정이 아니라 새 row 추가로 처리할 수 있게 설계했다.

## 전체 로직 흐름

### 프론트 -> order -> payment -> Toss

1. 프론트가 `order`에 주문 생성 요청
2. `order`가 `PENDING` 주문 생성 후 `orderId` 반환
3. 프론트가 Toss 위젯 호출
4. Toss success 결과로 `paymentKey`, `orderId`, `amount` 확보
5. 프론트가 `POST /api/payments/card/confirm` 호출
6. payment가 먼저 `order`에 `orderId`, `amount`로 검증 요청
7. 검증 성공 시 `card_transaction`에 `PENDING PAYMENT` row들을 저장
8. payment가 Toss confirm 호출
9. 성공하면 `card_transaction`을 `SUCCESS`로 반영
10. order에게 payment confirm 성공 이벤트 발행
11. `transactionGroupId` 포함 응답을 프론트에 반환
