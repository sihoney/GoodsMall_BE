# 주문 환불 구현 가이드

## 1. 문서 목적
이 문서는 Payment 모듈에 구현된 주문 환불 기능을 팀원이 빠르게 이해할 수 있도록 정리한 공유 문서입니다.

이 문서에서 설명하는 내용은 아래와 같습니다.

| 항목 | 설명 |
|---|---|
| 왜 이렇게 설계했는가 | Payment와 Order의 책임 분리, 원장 중심 설계 |
| 어떤 API로 통신하는가 | 주문 환불 요청 API 계약 |
| 내부에서 어떻게 처리되는가 | 서비스 흐름, 상태 전이, 트랜잭션 경계 |
| 어떤 데이터가 저장되는가 | `payment_refund`, `payment_refund_item`, 기존 거래 이력과의 관계 |
| escrow는 어떻게 연결되는가 | `orderItemId` 기준 차감 및 상태 전이 방식 |

---

## 2. 설계 배경과 의도

### 2.1 Payment가 책임지는 것

주문 환불은 "주문 상태 판단"과 "금전 환불 실행"이 섞이기 쉬운 기능입니다.
이 구현에서는 Payment가 금액 원장과 결제수단 실행만 책임지도록 경계를 두었습니다.

| 구분 | Order 책임 | Payment 책임 |
|---|---|---|
| 취소 가능 여부 판단 | O | X |
| 수량/배송/주문 상태 판단 | O | X |
| 환불 금액 계산 | O | X |
| 환불 원장 저장 | X | O |
| 카드 PG 취소 호출 | X | O |
| 예치금 환불 반영 | X | O |
| 거래 이력 기록 | X | O |

### 2.2 금액 중심 구조로 정리한 이유

이번 구현에서는 Payment가 수량을 직접 다루지 않습니다.
Order가 환불 대상과 금액을 결정하고, Payment는 전달받은 금액 기준으로 원장과 실행만 담당합니다.

그래서 환불 입력은 아래 구조를 사용합니다.

| 필드 | 의미 |
|---|---|
| `orderItemId` | 어떤 주문 항목을 환불하는지 식별 |
| `refundAmount` | 해당 항목에 대해 얼마를 환불할지 |

이 방식으로 정리한 이유는 아래와 같습니다.

1. Payment는 금액 실행에 집중할 수 있습니다.
2. Order 도메인의 수량/부분취소 정책이 Payment 안으로 새어 들어오지 않습니다.
3. escrow와 카드 원결제 모두 `orderItemId` 기준으로 맞출 수 있습니다.

## 4. 데이터 모델

### 4.1 `payment_refund`

주문 환불 1건의 헤더입니다.
환불 단위의 상태 전이와 멱등 키를 관리합니다.

| 컬럼 | 설명 |
|---|---|
| `refund_id` | 환불 식별자 |
| `order_cancel_request_id` | 멱등 키 |
| `order_id` | 주문 식별자 |
| `buyer_member_id` | 구매자 식별자 |
| `refund_type` | `FULL`, `PARTIAL` |
| `payment_method` | `WALLET`, `CARD`, `MIXED` |
| `total_refund_amount` | 총 환불 금액 |
| `refund_status` | `REQUESTED`, `PROCESSING`, `SUCCEEDED`, `FAILED` |
| `refund_reason` | 환불 사유 |
| `completed_at` | 성공 완료 시각 |
| `failed_at` | 실패 시각 |

### 4.2 `payment_refund_item`

주문 항목 단위 환불 상세입니다.
환불 헤더 아래에 여러 건이 저장됩니다.

| 컬럼 | 설명 |
|---|---|
| `refund_item_id` | 환불 상세 식별자 |
| `refund_id` | 환불 헤더 FK |
| `order_item_id` | 환불 대상 주문 항목 |
| `refund_amount` | 항목별 환불 금액 |
| `status` | `REQUESTED`, `SUCCEEDED`, `FAILED` |
| `failure_reason` | 상세 실패 사유 |

### 4.3 `escrow`

정산 전 보류 중인 금액입니다.
이번 구조에서는 `orderItemId` 단위로 식별할 수 있게 확장했습니다.

| 컬럼 | 설명 |
|---|---|
| `reference_id` | 외부 기준 식별자, 현재는 주로 `orderItemId` |
| `reference_type` | `ORDER`, `ORDER_ITEM` |
| `amount` | 현재 남은 보류 금액 |
| `original_amount` | 최초 보류 금액 |
| `refunded_amount` | 누적 환불 금액 |
| `escrow_status` | `HELD`, `RELEASED`, `REFUNDED` |

### 4.4 기존 거래 이력과의 연결

새 환불 테이블을 만들더라도, 실제 거래 이력은 기존 원장을 유지하도록 설계했습니다.

| 테이블 | 연결 방식 |
|---|---|
| `wallet_transaction` | `transaction_type=REFUND`, `reference_id=refund_id` |
| `card_transaction` | CANCEL 거래를 별도 row로 저장하고 `related_transaction_id`로 원결제 연결 |

## 5. API 명세

### 5.1 Endpoint

| 항목 | 값 |
|---|---|
| Method | `POST` |
| Path | `/api/payments/refunds` |
| 설명 | 주문 환불 요청 |

### 5.2 Request 필드

| 필드 | 타입 | 필수 | 설명 |
|---|---|---|---|
| `orderId` | UUID | Y | 주문 식별자 |
| `buyerMemberId` | UUID | Y | 구매자 식별자 |
| `orderCancelRequestId` | UUID | Y | 멱등 키 |
| `refundType` | ENUM | Y | `FULL`, `PARTIAL` |
| `paymentMethod` | ENUM | Y | `WALLET`, `CARD`, `MIXED` |
| `reason` | String | 조건부 | 현재 카드 환불 시 필수 |
| `items` | Array | Y | 환불 대상 목록 |

요청 계약 전제:

| 항목 | 전제 |
|---|---|
| `items.orderItemId` | 요청 1건 내에서 중복 없이 1회씩 전달 |
| 중복 전달 시 처리 기준 | Order에서 사전 정규화 후 전달, Payment는 금액 실행/원장 저장에 집중 |

결제수단 판별 기준:

| 항목 | 기준 |
|---|---|
| `paymentMethod` | Order가 확정해서 전달 (`WALLET`, `CARD`, `MIXED`) |
| Payment 역할 | 전달된 `paymentMethod` 기준으로 환불 실행 |
| 주의사항 | 총 환불 금액만으로는 혼합 결제를 안정적으로 추론할 수 없음 |

서비스 간 계약(책임 기준):

| 책임 | Order | Payment |
|---|---|---|
| 환불 가능 여부 판단 | 책임 | 비책임 |
| 환불 대상/금액 계산 | 책임 | 비책임 |
| `paymentMethod` 확정 전달 | 책임 | 비책임 |
| 환불 원장 기록 | 비책임 | 책임 |
| 결제수단 환불 실행(wallet/card) | 비책임 | 책임 |
| 환불 결과 상태 기록 | 비책임 | 책임 |

계약 문장:

1. Order는 취소/환불 정책을 적용해 `paymentMethod`, `items(orderItemId, refundAmount)`를 확정해 Payment에 전달한다.
2. Payment는 전달받은 계약을 기준으로 금액 실행(wallet/card), 환불 원장(`payment_refund`, `payment_refund_item`) 기록, 거래 이력 기록을 수행한다.
3. Payment는 환불 금액 산정 책임을 갖지 않으며, 전달된 금액의 실행/정합성 검증 책임만 가진다.
4. MIXED 환불에서 결제수단 분해 기준은 Order 책임이며, Payment는 전달된 기준대로 실행한다.

### 5.5 MIXED Request 예시

```json
{
  "orderId": "f1a4d3f1-0000-0000-0000-000000000001",
  "buyerMemberId": "7f7ad7b2-0000-0000-0000-000000000001",
  "orderCancelRequestId": "2b5fb48e-0000-0000-0000-000000000002",
  "refundType": "PARTIAL",
  "paymentMethod": "MIXED",
  "reason": "부분 환불 요청",
  "items": [
    {
      "orderItemId": "9a90c6d9-0000-0000-0000-000000000001",
      "refundAmount": 10000
    },
    {
      "orderItemId": "9a90c6d9-0000-0000-0000-000000000002",
      "refundAmount": 8000
    }
  ]
}
```

MIXED 요청 계약(필수):

| 항목 | 계약 |
|---|---|
| `paymentMethod` | 반드시 `MIXED`로 전달 |
| `items` | 환불 대상 `orderItemId`를 유니크하게 전달 |
| `reason` | 카드 취소가 포함되므로 필수 전달 |
| 금액 기준 | `sum(items.refundAmount)`는 Order가 확정한 총 환불 금액과 일치해야 함 |

MIXED 요청 계약(권장 확장):

| 항목 | 목적 |
|---|---|
| `walletRefundAmount` | 예치금 환불 실행 금액을 명시해 책임 경계 명확화 |
| `cardRefundAmount` | 카드 취소 실행 금액을 명시해 PG 실행 정합성 강화 |

권장 계약 문장:

1. Order는 MIXED 환불 시 `walletRefundAmount`와 `cardRefundAmount`를 함께 전달한다.
2. Payment는 전달된 수단별 환불 금액을 그대로 실행하고, 실행 결과를 환불 원장과 거래 이력에 기록한다.

## 6. 처리 흐름

### 6.1 전체 환불 흐름

```text
Order
  -> POST /api/payments/refunds
  -> PaymentController.requestPaymentRefund(...)
  -> PaymentRefundService.requestRefund(...)
     1. 요청 검증
     2. orderCancelRequestId 멱등 조회
     3. payment_refund / payment_refund_item 저장
     4. REQUESTED -> PROCESSING 전이
     5. 결제수단별 환불 실행
     6. 성공/실패 상태 저장
     7. 결과 응답 반환
```

### 6.2 WALLET 환불 흐름

```text
refund request
  -> buyer wallet 조회
  -> totalRefundAmount 만큼 wallet 잔액 증가
  -> wallet_transaction REFUND 저장
  -> refund item의 orderItemId 목록으로 HELD escrow 조회
  -> orderItem별 refundAmount 차감
  -> 차감 후 escrow 잔액이 0이면 REFUNDED
```

### 6.3 CARD 환불 흐름

```text
refund request
  -> orderItemId 기준 원결제(PAYMENT/SUCCESS) 조회
  -> buyerMemberId 검증
  -> 원결제들의 paymentKey 일치 검증
  -> Toss cancel(paymentKey, reason, totalRefundAmount)
  -> CANCEL card_transaction 생성 및 저장
  -> refund 성공 상태 저장
```

## 7. 상태 전이

### 7.1 환불 헤더 상태

| 상태 | 의미 |
|---|---|
| `REQUESTED` | 환불 원장 저장 직후 |
| `PROCESSING` | 실제 환불 실행 시작 |
| `SUCCEEDED` | 환불 성공 |
| `FAILED` | 환불 실패 |

### 7.2 환불 상세 상태

| 상태 | 의미 |
|---|---|
| `REQUESTED` | 상세 저장 직후 |
| `SUCCEEDED` | 상세 환불 성공 |
| `FAILED` | 상세 환불 실패 |

### 7.3 escrow 상태

| 상태 | 의미 |
|---|---|
| `HELD` | 정산 전 보류 중 |
| `RELEASED` | 구매확정 후 정산 대상 전환 |
| `REFUNDED` | 환불로 인해 보류 잔액이 0이 된 상태 |

## 8. 멱등성과 정합성

### 8.1 멱등 처리 방식

같은 환불 요청이 네트워크 재시도 등으로 반복될 수 있기 때문에,
환불 API는 `orderCancelRequestId`를 멱등 키로 사용합니다.

```text
same orderCancelRequestId
  -> 기존 payment_refund 조회
  -> 기존 payment_refund_item 조회
  -> 기존 결과 그대로 반환
```

### 8.2 현재 서비스가 맞추는 정합성 기준

| 항목 | 규칙 |
|---|---|
| 총 환불 금액 | `sum(items.refundAmount)` |
| item 식별 규칙 | 요청 내 `orderItemId`는 유니크 전제 |
| wallet 환불 금액 | `wallet_transaction.amount = totalRefundAmount` |
| card 환불 금액 | Toss 취소 금액과 요청 금액 일치 |
| escrow 환불 금액 | item별 차감 합계와 총 환불 금액 일치 |
| buyer 정합성 | wallet, escrow, card 원결제의 buyer가 모두 일치해야 함 |

## 9. 이벤트와 외부 연동

### 9.1 외부 연동

| 대상 | 사용 위치 | 목적 |
|---|---|---|
| Toss Payment Gateway | 카드 환불 실행 시 | 카드 취소 요청 |

현재 주문 환불 기능 자체에서 별도의 Kafka 이벤트를 직접 발행하지는 않습니다.
환불의 실행 결과는 Payment 내부 원장과 거래 이력으로 우선 기록되도록 구현했습니다.

### 9.2 관련 이벤트 흐름과의 관계

환불 기능은 Payment 내부에서 동작하지만,
기존 주문 결제/escrow 해제 흐름과 같은 맥락 위에 올라가 있습니다.

| 흐름 | 관련 서비스 | 설명 |
|---|---|---|
| 예치금 주문 결제 | `OrderPaymentService` | buyer wallet 차감 + orderItem 단위 escrow 생성 |
| 카드 결제 확정 | `CardPaymentConfirmService` | card transaction 승인 저장 |
| 구매확정 후 escrow 해제 | `EscrowReleaseService` | escrow를 RELEASED로 전이하고 후속 이벤트 발행 |
| 주문 환불 | `PaymentRefundService` | refund 원장 저장 + wallet/card 환불 실행 |

## 10. 구현 포인트 요약

### 10.1 왜 `orderItemId` 기준인가

`orderItemId` 기준을 사용하면 아래가 하나의 축으로 맞춰집니다.

1. 환불 요청 입력
2. 카드 원결제 식별
3. escrow 차감 대상
4. 환불 상세 원장

즉, "어떤 주문 항목을 얼마나 환불했는지"를 Payment 내부에서 가장 일관되게 추적할 수 있습니다.

### 10.2 왜 서비스 본문을 private 메서드로 쪼갰는가

`requestRefund(...)`는 오케스트레이션 역할만 남기고,
검증, 금액 합산, wallet 처리, card 처리, 결과 매핑 등을 private 메서드로 분리했습니다.

이렇게 한 이유는 아래와 같습니다.

| 이유 | 설명 |
|---|---|
| 트랜잭션 흐름 가독성 확보 | 본문에서 상태 전이와 분기만 보이게 하기 위함 |
| 결제수단별 로직 분리 | wallet/card 책임을 명확히 나누기 위함 |
| 유지보수성 향상 | 검증/매핑/저장 로직을 개별적으로 읽기 쉽게 하기 위함 |

### 10.3 왜 환불 이력은 새 테이블 + 기존 거래 원장을 함께 쓰는가

환불은 "주문 환불 유스케이스"와 "실제 돈의 흐름"이 모두 필요합니다.
그래서 아래 두 층을 분리해서 기록했습니다.

| 층 | 역할 |
|---|---|
| `payment_refund`, `payment_refund_item` | 환불 요청과 결과를 도메인 단위로 기록 |
| `wallet_transaction`, `card_transaction` | 실제 자금 이동 이력을 기록 |

## 12. 개발 고려 사항 (MIXED 결제/환불)

### 12.1 현재 결제 진입 구조 기준 해석

현재 Payment 결제 경로는 기능적으로 아래처럼 분리되어 있습니다.

| 경로 | API | 성격 |
|---|---|---|
| 예치금 주문 결제 | `POST /api/payments/orders` | wallet 기반 주문 결제 |
| 카드 결제 확정 | `POST /api/payments/card/confirm` | PG 승인 기반 카드 결제 |

이 구조에서는 "예치금 일부 사용 후 카드 결제"를 2단계로 처리하는 시나리오가 자연스럽습니다.

### 12.2 팀 합의 전제(권장)

| 항목 | 권장 기준 |
|---|---|
| MIXED 여부 판단 주체 | Order |
| MIXED 판단 시점 | 결제 완료 후 Order가 최종 결제 구성(예치금/카드)을 확정한 시점 |
| Payment 입력 | Order가 확정한 `paymentMethod`와 환불 금액 계약을 그대로 전달 |
| Payment 책임 | 전달된 계약의 실행/정합성 검증/원장 기록 |

### 12.3 실무 계약 문장(권장안)

1. 프론트는 결제 단계에서 예치금 사용 금액과 카드 결제 금액을 결정한다.
2. Order는 결제 완료 이후 최종 결제 구성을 기준으로 주문의 결제수단을 `WALLET`, `CARD`, `MIXED` 중 하나로 확정한다.
3. 환불 요청 시 Order는 확정된 결제수단(`paymentMethod`)과 환불 계약(`items`, 필요 시 수단별 환불 금액)을 Payment에 전달한다.
4. Payment는 전달받은 계약을 실행하고 환불 원장과 거래 이력만 기록한다.

### 12.4 오버엔지니어링 방지 기준

| 상황 | 권장 |
|---|---|
| MIXED 결제가 실제로 아직 없을 때 | Order에서 `MIXED` 요청 자체를 보내지 않도록 정책 차단 |
| MIXED 결제가 실제로 도입될 때 | Order 계약에 수단별 환불 금액(`walletRefundAmount`, `cardRefundAmount`)을 명시해 도입 |

정리하면, MIXED 환불은 "필요해졌을 때 계약을 명확히 두고 켠다"가 가장 안전합니다.

### 12.5 MIXED 도입 시 필요한 구현 항목

`예치금 선결제 + 카드 후결제` 방식으로 MIXED를 도입한다면, 아래 항목을 기준으로 정리하면 됩니다.

| 구분 | 항목 | 책임 | 상태(현재) |
|---|---|---|---|
| 결제 오케스트레이션 | 예치금 결제 성공 후 카드 결제로 이어지는 주문 결제 흐름 확정 | Order | 추가 필요 |
| 결제 결과 기록 | 주문 단위 최종 결제수단(`MIXED`)과 수단별 사용 금액 기록 | Order | 추가 필요 |
| 환불 계약 | 환불 요청 시 `paymentMethod=MIXED` + 수단별 환불 금액 전달 | Order | 추가 필요 |
| 환불 실행/원장 기록 | 전달된 계약 기준 wallet/card 환불 실행 및 원장 기록 | Payment | 구현됨 |

해석:
1. Payment의 MIXED 환불 실행 자체는 이미 구현되어 있습니다.
2. 실사용 가능한 MIXED를 완성하려면 Order 쪽 결제 흐름/계약 확정이 추가되어야 합니다.

권장 계약 문장:

1. Order는 MIXED 결제 도입 시 주문의 최종 결제 구성을 단일 소스로 관리한다.
2. Payment는 해당 결제 구성을 조회/추론하지 않고, Order가 전달한 환불 계약만 실행한다.
3. MIXED 도입 전까지는 `MIXED` 환불 요청을 차단해 단일 수단 환불만 운영한다.
